### 1. Claude 에이전트가 잠자는 동안 스스로 배운다 — **Dreaming**

**title** `Claude Managed Agents에 "Dreaming"이 왔다`

**hook**
어제 에이전트가 같은 실수를 세 번 반복했다면, 오늘 밤 Dreaming이 그 패턴을 찾아내고, 내일 아침엔 조금 더 나은 에이전트로 깨어난다. 개발자가 아무것도 안 해도.

**source** [claude.com/blog/new-in-claude-managed-agents](https://claude.com/blog/new-in-claude-managed-agents) — 5월 6일 발표, 어제 막 나온 소식

**why_now**
Dreaming(Research Preview)은 에이전트의 과거 세션과 메모리 스토어를 주기적으로 검토해 반복 실수·공통 워크플로·팀 선호도를 자동 추출한다. 함께 공개된 **Outcomes**(rubric 기반 자동 평가)와 **Multiagent Orchestration**(리드 에이전트가 서브에이전트에게 작업을 병렬 위임)까지 합치면, Paper Company의 에이전트 루프를 아예 다른 레이어로 끌어올릴 수 있다.

**why_fit**
Paper Company에서 에이전트한테 반복적으로 같은 피드백을 주고 있다면, Dreaming은 그걸 없애는 첫 번째 진짜 도구다. "에이전트가 스스로 나아진다"는 건 SaaS product의 retention curve를 바꾸는 이야기이기도 하다.

**next_action**
> 블로그 포스트 읽기 (15분) → 내가 Paper Company 에이전트에게 반복적으로 교정해주는 패턴 3개 적어보기 (10분) → Dreaming이 그 패턴을 자동으로 잡을 수 있을지 판단하기

**expansion**
- Outcomes의 rubric을 Paper Company 에이전트 평가 기준으로 설계 → 품질 관리 파이프라인 자동화
- Multiagent Orchestration으로 "리드 에이전트 + 전문 서브에이전트" 구조 첫 설계

**exploration_path**
Claude Agent SDK 최신 동향 검색 → 5월 7일 발표 발견 → Dreaming이 Paper Company 에이전트 루프에 직결되는 개념임을 확인 → 세 가지 기능(Dreaming / Outcomes / Multiagent) 모두 실제로 쓸 수 있는 구조인지 검증

**why_fit**
money-making automation의 현재 가장 극단적인 실제 사례. 이 구조를 분해하면 Paper Company에서 작은 버전을 먼저 만들어볼 수 있는 청사진이 나온다.

**next_action**

### 3. Claude가 자연어로 내 로컬 데이터를 쿼리한다 — **DuckDB + MCP**

**title** `DuckDB에 MCP 서버가 붙었다 — AI가 SQL 없이 내 데이터를 직접 읽는다`

**hook**
"지난달 에이전트가 가장 많이 실패한 액션이 뭐야?"라고 Claude에게 물으면, SQL을 짤 필요 없이 바로 답이 나온다. DuckDB가 MCP 서버를 내장했고, Claude를 직접 연결할 수 있다. 서버도, 클라우드도 필요 없다.

**source** [programming-helper.com — DuckDB 2026](https://www.programming-helper.com/tech/duckdb-2026-in-process-analytics-python) / [MotherDuck DuckDB Ecosystem Newsletter Mar 2026](https://motherduck.com/blog/duckdb-ecosystem-newsletter-march-2026/)

**why_now**
DuckDB 1.5.2 + DuckLake 1.0이 최근 릴리즈됐다. MCP 연동으로 Claude, GPT-4o, 로컬 모델이 자연어로 데이터베이스에 직접 접근할 수 있게 됐다. Parquet 파일만 있으면 된다.

**why_fit**
Paper Company 에이전트 로그를 DuckDB에 쌓으면, Claude가 자연어로 "어제 무슨 실수를 했나"를 직접 분석하는 파이프라인이 만들어진다. 별도 데이터 인프라 없이.

**next_action**
> `pip install duckdb` (5분) → 샘플 CSV 또는 로그 파일 하나 열어서 쿼리 3개 날려보기 (20분) → MCP 연결 문서 북마크

**expansion**
- Paper Company 에이전트 로그 → DuckDB 저장 → Claude가 자연어로 분석하는 파이프라인 구축
- DuckLake Time-travel 쿼리로 "이틀 전 에이전트 상태"를 언제든 재현할 수 있는 디버깅 시스템

**exploration_path**
Backend open source 2026 탐색 → DuckDB가 MCP를 내장했다는 사실 발견 → "이건 Paper Company 에이전트 분석에 오늘 바로 붙일 수 있다"는 판단 → 구체적인 연동 구조 확인

### 4. Express보다 9배 빠른 TypeScript 백엔드 — **Encore.ts**

**title** `Encore.ts: Rust 런타임 + 인프라 자동 프로비저닝 = 솔로 파운더의 백엔드 스택`

**hook**
TypeScript로 `new Service()`를 선언하는 순간, 데이터베이스·Pub/Sub·cron job이 로컬과 AWS/GCP에 자동으로 만들어진다. Zod schema도, 수동 validation도 없다. Rust 런타임이 붙어 Express보다 9배 빠르다.

**source** [encore.cloud — Best TypeScript Backend Frameworks 2026](https://encore.cloud/resources/best-typescript-backend-frameworks-2026) / [encore.dev — Express vs Encore](https://encore.dev/articles/expressjs-vs-encore)

**why_now**
2026년 TypeScript 백엔드 프레임워크 비교에서 Encore.ts가 1위로 꼽히고, AI-assisted development에 가장 잘 맞는 프레임워크로도 선정됐다.

**why_fit**
Paper Company 백엔드를 확장할 때, 인프라를 수동으로 셋업하는 시간을 없애준다. 코드에서 인프라를 선언하면 끝이라는 DX는 솔로 파운더 스택으로 최적화됐다.

**next_action**
> [encore.dev](https://encore.dev) 접속 → TypeScript quickstart 따라서 첫 서비스 30분 안에 로컬 배포까지

**expansion**
- Paper Company의 에이전트 워크플로우를 Encore.ts로 리팩토링하면 인프라 비용과 복잡도를 동시에 축소
- 블로그 글: "Encore.ts로 갈아탄 이유 — 인프라 선언만 남기고 나머지는 다 없앴다"

**exploration_path**
TypeScript backend 2026 검색 → Encore.ts 발견 → "Rust 런타임 + Express 9배 성능"이라는 조합이 흥미로워서 파고듦 → AI-assisted development 최적화 프레임워크라는 포지셔닝 확인
